In-Wheel Motor Drive System

Project Period

Jan 2025 - Mar 2025 (3 Months)

My Role

Designed safety circuit board and sole developer of STM32 firmware

Tech Stack

C++, STM32(HAL), CAN Bus, KiCad (PCB Design)

Main Devices

MD400T Driver, MDH250 In-Wheel Motors, MRE-RR1R E-Stop Switch

Project Overview

Developed a control interface for a dual **MDH250 high-torque in-wheel motor** platform (commonly used in mobile robots/AGVs) driven by an **MD400T 2-axis motor driver**, utilizing an STM32 MCU via **CAN bus** communication to monitor and synchronize velocities in real-time.

Dual-Channel Safety Circuit Design

To prevent accidents from firmware/hardware malfunctions, I designed an emergency stop circuit integrating a physical **MRE-RR1R emergency switch**. Aside from software stop flags, triggering the E-Stop instantly cuts the main power relays and cuts off the STO (Safe Torque Off) terminal directly at the hardware layer, ensuring absolute safety.

Troubleshooting

Problem: Sudden braking or downhill travel generated high back-EMF, causing spikes in the main DC link voltage. This triggered Over-Voltage faults on the motor controller and reset the CPU.

Solution: 1. Added a high-capacity bypass capacitor bank and reverse-current protection diodes to the power entry stage of the circuit board.
2. Implemented an S-Curve acceleration and deceleration profile in the STM32 firmware to mitigate torque and back-EMF spikes by 40% during transitions.

Vision-based Object Detection

Project Overview

Built a real-time multi-object detection pipeline based on camera streams, enabling mobile robots to detect and localize obstacles in their surrounding environment.

Intel OpenVINO Edge Acceleration

Mobile robots operating under battery power often lack heavy desktop GPUs. To run deep learning models efficiently, I exported the trained PyTorch models to **OpenVINO IR format** and applied **FP16 quantization**, accelerating the frame rate (FPS) by **2.4x** on low-power Intel CPU cores with less than 1% loss in detection accuracy.

Troubleshooting

Problem: Running high-resolution image preprocessing (denoising, resizing) and model inference on a single thread caused severe CPU bottlenecks. This created a frame buffer backlog, causing visual latency to accumulate to over 0.5 seconds.

Solution: Redesigned the code into a multi-threaded **Asynchronous Pipeline**. Built a thread-safe frame queue: a dedicated capture thread feeds input frames into the queue, while the inference thread asynchronously pops and feeds them to the OpenVINO accelerator. This decoupled architecture eliminated frame drops and ensured a stable 30 FPS.
